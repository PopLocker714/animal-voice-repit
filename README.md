# Повтори за животным 🐑

Веб-игра (PWA) для детей: игрок вводит имя, жмёт «Начать игру», и ему по очереди
в случайном порядке показываются 20 животных. Для каждого: слушает звук →
повторяет в микрофон → видит, насколько похоже (в %). Голос записывается и
сохраняется на сервере — его можно прослушать позже.

## Игровой поток

1. **Имя** + выбор режима:
   - **🌍 Онлайн** — игрок создаётся на сервере, результаты идут в общую
     живую таблицу, записи хранятся на сервере.
   - **📴 Локально** — никакого сервера и таблицы; всё (включая записи голоса
     в IndexedDB) хранится только на устройстве. Работает без интернета.
2. **20 раундов** в случайном порядке (`src/game/session.ts` — Fisher-Yates,
   прогресс в localStorage, переживает обновление страницы).
3. **Результаты**: средний %, сумма очков, список всех животных с прослушиванием
   своих записей. Онлайн дополнительно показывает живую таблицу с подсветкой
   твоего места.

Имя запоминается и подставляется в поле при следующей игре (можно изменить).
Хранилище записей в локальном режиме — `src/game/localStore.ts` (IndexedDB).

## Админка `/admin`

Тот же домен, путь `/admin` (без защиты — игра для детей):

- **По игрокам**: таблица всех игроков по сумме очков (лучший — первый, 🥇).
  Клик по игроку → его результаты + прослушивание записей.
- **По животным**: для каждого из 20 животных — рекордсмен (👑 кто набрал
  лучший %). Клик по строке открывает профиль этого игрока.
- «Сбросить игру» — стирает всех игроков и все записи.

Оба вида обновляются в реальном времени (WebSocket).

## Бэкенд

Bun-сервер (`server.ts` + `server/`): API + раздача статики, без внешних
зависимостей. Хранилище — **SQLite** (`bun:sqlite`, `server/db.ts`); записи
голоса лежат файлами в `DATA_DIR/recordings/`. Оценка похожести по-прежнему
считается **в браузере** (MFCC+DTW), на сервер уходит только число и аудио.

Эндпоинты: `POST /api/users`, `POST /api/results` (multipart с аудио),
`GET /api/users/:id/results`, `GET /api/admin/users`, `GET /api/admin/users/:id`,
`POST /api/admin/reset`, `GET /api/recordings/:id`.

## Как считается похожесть

1. Записи (эталон животного и попытка пользователя) приводятся к 16 кГц моно,
   нормализуются по громкости, обрезается тишина.
2. Из каждой извлекаются **MFCC** (мел-кепстральные коэффициенты) — описание тембра
   по кадрам ~32 мс с шагом ~16 мс (`src/audio/features.ts`).
3. Две последовательности MFCC выравниваются по времени через **DTW**
   (Dynamic Time Warping, `src/audio/dtw.ts`) — пользователь может «блеять»
   медленнее или быстрее.
4. Нормированное DTW-расстояние переводится в 0–100 % через сигмоиду
   (`src/audio/score.ts`). Сырое расстояние пишется в консоль для калибровки.

Сравнение считается в браузере; на сервер уходит только результат и запись.

## Запуск (разработка)

Фронт и API запускаются отдельно (Vite проксирует `/api` на сервер):

```bash
bun install
bun server.ts        # API + SQLite на :3000 (терминал 1)
bun run dev          # фронт на :5173, проксирует /api -> :3000 (терминал 2)
```

Открой http://localhost:5173 — микрофон работает на `localhost` без https.
Данные пишутся в `./data` (переопределяется через `DATA_DIR`).

### Проверка с телефона

Микрофону нужен защищённый контекст. Для теста на телефоне в той же сети:
задеплой на Cloudflare Pages (`bun run build` → папка `dist/`) или подними https
локально. По http по LAN-адресу микрофон не включится — это ограничение браузера.

## Звуки животных

Аудио (CC0) кладётся в `public/sounds/` — см. `public/sounds/README.md`.
Без файлов приложение всё равно запускается и даёт записывать.

## Калибровка оценки

После нескольких попыток смотри в консоль значения `DTW distance` и подгони
`MIDPOINT` / `STEEPNESS` в `src/audio/score.ts`, чтобы точное повторение давало
~90 %, а случайный звук — ~20 %.

## PWA — установка и кеш

Приложение — устанавливаемый PWA (`vite-plugin-pwa` + Workbox). Сервис-воркер
кеширует аппку и все 20 звуков, поэтому интерфейс и звуки грузятся мгновенно и
работают офлайн. Сохранение результатов и запись голоса требуют сервера (это
сетевые вызовы `/api`). Обновления версии применяются автоматически.

- Установка: в браузере «Установить приложение» / «На экран Домой».
- Настройка манифеста и прекеша — в `vite.config.ts` (блок `VitePWA`).
- Иконки: `public/icon-{192,512}.png`, `icon-maskable-512.png`, `apple-touch-icon.png`.

## Деплой через Docker Compose

Bun собирает фронт и сам раздаёт его + API (без nginx).

```bash
docker compose up -d --build   # собрать и запустить → http://localhost:8082
docker compose down            # остановить
```

- `Dockerfile` — multi-stage: Bun собирает `dist/`, финальный образ запускает
  `bun server.ts` (API + статика). BuildKit cache mount ускоряет `bun install`
  на повторных сборках; в образе только `vite build` (без `tsc`).
- Контейнер слушает порт **3000**; compose пробрасывает `8082:3000`.
- **Данные** (SQLite + записи голоса) лежат в `/app/data` и сохраняются в
  именованном томе `app-data` — переживают пересборку/редеплой.

### Dokploy

Build Type → **Dockerfile**, Container Port → **3000**. Traefik сам выдаёт
домен с HTTPS (нужен для микрофона). Для сохранности данных смонтируй
постоянный том на `/app/data` в настройках сервиса.

### Медленный / слабый сервер (сборка «висит»)

Тайп-чек (`tsc`) в образе **не запускается** — собирается только `vite build`,
поэтому нагрузка от типов нулевая. Если сборка всё равно зависает — это почти
всегда нехватка памяти: установка пакетов и сборка уходят в своп и сервер
начинает «тормозить до зависания». Лечится свапом:

```bash
sudo fallocate -l 2G /swapfile   # 2 ГБ swap (хватает с запасом)
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab   # чтобы пережил перезагрузку
free -h                          # проверить, что swap появился
```

Сам `vite build` уже облегчён (без `tsc`, без terser, без подсчёта gzip-размеров
— см. `build` в `vite.config.ts`).

**Если сервер совсем слабый** — собирай `dist/` у себя/в CI и деплой готовую
статику, не нагружая сервер сборкой:

```bash
bun install && bun run build     # локально → dist/
```

а на сервере подними `dist` + `server.ts` без стадии сборки (можно отдельным
compose, монтируя `./dist`, `./server.ts`, `./server/` в образ `oven/bun:1-alpine`).

### Микрофон и HTTPS

Микрофон работает только в защищённом контексте (`localhost` или HTTPS). Для
доступа с телефона по сети поставь контейнер за HTTPS-реверс-прокси (Caddy,
Traefik, nginx + Let's Encrypt) или прокинь через туннель с TLS. По «голому»
`http://<LAN-IP>:8080` запись не включится — это ограничение браузера, не баг.
